# 视频链接格式网络调查分析报告

> 后台视频下载模块 - 兼容性与网络调查报告
> 生成日期：2026-06-20

## 一、调查概述

本报告针对主流视频网站使用的视频链接格式进行系统性网络调查，分析各类格式的技术特性、下载限制、浏览器环境依赖情况，并提供对应的下载策略。调查基于实际网络请求抓包、yt-dlp 项目资料、YouTube SABR 协议研究等多方面信息源。

## 二、视频链接格式分类

### 2.1 直接视频 URL（DIRECT_MP4 / DIRECT_AUDIO）

**技术特性：**
- URL 直接指向视频/音频文件（如 `.mp4`、`.webm`、`.m4a`、`.mp3`）
- 服务器返回完整文件，支持 HTTP Range 请求（断点续传）
- 无需特殊协议解析，直接下载即可

**典型示例：**
```
https://example.com/video.mp4
https://example.com/audio.m4a
```

**下载限制：**
- 部分服务器要求特定的 User-Agent 或 Referer 头
- 大文件可能受服务器超时限制
- 极少情况下需要 Cookie 认证

**浏览器环境依赖：** 无特殊依赖

**下载策略：**
- 使用 `UniversalDownloadManager` 直接下载
- 支持 HTTP Range 断点续传
- 优先级：60（视频）/ 50（音频）

**解决方案状态：** ✅ 完全支持

---

### 2.2 HLS 自适应流（HLS）

**技术特性：**
- 基于 HTTP 的自适应比特率流协议（HTTP Live Streaming）
- 由 Apple 提出，使用 `.m3u8` 播放列表文件描述分片
- 分片通常为 `.ts`（MPEG-TS）或 `.m4s`（fMP4）格式
- 支持 Master Playlist（多分辨率）和 Media Playlist（单分辨率分片列表）
- 支持加密（AES-128）

**典型示例：**
```
https://example.com/stream/index.m3u8
https://example.com/master.m3u8  (Master Playlist)
```

**URL 结构：**
```
Master Playlist:
  #EXT-X-STREAM-INF:RESOLUTION=1920x1080
  high/index.m3u8
  #EXT-X-STREAM-INF:RESOLUTION=1280x720
  mid/index.m3u8

Media Playlist:
  #EXTINF:10.0,
  segment_001.ts
  #EXTINF:10.0,
  segment_002.ts
  #EXT-X-ENDLIST
```

**下载限制：**
- 需要解析 m3u8 播放列表获取分片 URL
- 分片 URL 可能是相对路径（需基于 base URL 解析）
- 加密流需要获取密钥（URI 在 `#EXT-X-KEY` 标签中）
- 分片数量可能动态变化（直播流）

**浏览器环境依赖：**
- WebView 的 `shouldInterceptRequest` 可捕获 m3u8 请求
- MSE（Media Source Extensions）播放的 HLS 无法直接拦截分片请求
- 需要通过 JS 注入拦截 fetch/XHR 请求

**下载策略：**
- 使用 `SegmentMerger.downloadAndMergeHls()` 下载并合并
- 自动选择 Master Playlist 中最高分辨率的 variant
- 支持相对 URL 解析
- 分片顺序合并为单一文件
- 优先级：100（最高）

**解决方案状态：** ✅ 完全支持（已测试验证）

---

### 2.3 DASH 自适应流（DASH）

**技术特性：**
- 基于 HTTP 的动态自适应流协议（Dynamic Adaptive Streaming over HTTP）
- 使用 `.mpd`（Media Presentation Description）XML 文件描述流
- 分片通常为 `.m4s`（fMP4）格式，包含 init segment + media segments
- 支持 DRM（Widevine、PlayReady）
- YouTube、Netflix 等主流平台使用

**典型示例：**
```
https://example.com/stream/manifest.mpd
```

**URL 结构：**
```xml
<MPD>
  <Period>
    <AdaptationSet mimeType="video/mp4">
      <Representation bandwidth="5000000">
        <BaseURL>video_1080p/</BaseURL>
        <SegmentList>
          <Initialization sourceURL="init.m4s"/>
          <SegmentURL media="segment_1.m4s"/>
          <SegmentURL media="segment_2.m4s"/>
        </SegmentList>
      </Representation>
    </AdaptationSet>
  </Period>
</MPD>
```

**下载限制：**
- 需要解析 MPD XML 获取分片信息
- 视频和音频通常是分离的（需分别下载后合并）
- DRM 加密流无法直接下载
- 分片 URL 可能使用模板（`$Number$`、`$Time$`）

**浏览器环境依赖：**
- 与 HLS 类似，MSE 播放的 DASH 无法直接拦截
- 需要通过 JS 注入拦截请求

**下载策略：**
- 使用 `SegmentMerger` 下载分片并合并
- 需要分别处理视频和音频流
- 优先级：100（与 HLS 相同）

**解决方案状态：** ⚠️ 基础支持（未完整实现 MPD XML 解析，当前按 HLS 方式处理）

---

### 2.4 YouTube googlevideo 流（GOOGLEVIDEO）

**技术特性：**
- YouTube 专用的视频分发协议，通过 `googlevideo.com/videoplayback` URL 提供
- 传统格式使用 `itag` 参数标识视频格式/分辨率
- 支持 HTTP Range 请求分段下载
- URL 包含签名参数（`sig`/`sparams`），有时效性

**传统 itag 格式示例：**
```
https://rr5---sn-xxx.googlevideo.com/videoplayback?
  expire=...&ei=...&ip=...&id=...&itag=137&source=youtube&
  requiressl=yes&mime=video/mp4&...&sig=...
```

**itag 常见值：**
| itag | 格式 | 分辨率 | 编码 |
|------|------|--------|------|
| 137  | MP4  | 1080p  | H.264 |
| 248  | WebM | 1080p  | VP9  |
| 136  | MP4  | 720p   | H.264 |
| 140  | M4A | 音频   | AAC  |
| 251  | WebM| 音频   | Opus |

**下载限制：**
- URL 有时效性（`expire` 参数，通常 6 小时）
- 需要特定的请求头（User-Agent、Referer、Origin）
- 缺少请求头会返回 403 Forbidden
- 视频和音频流分离（DASH 模式）

**浏览器环境依赖：**
- 需要桌面版 User-Agent 获取完整格式
- 需要正确的 Referer（`https://www.youtube.com/`）

**下载策略：**
- 使用 `UniversalDownloadManager` 直接下载
- 自动添加 YouTube 专用请求头
- 优先级：80-90（视频流高优先级）

**解决方案状态：** ⚠️ 传统 itag 格式支持，但 YouTube 已全面迁移到 SABR 协议

---

### 2.5 YouTube SABR 协议流（ENCRYPTED）

**技术特性：**
- SABR（Server Adaptive Bit Rate）是 YouTube 2024-2025 年推出的新型流协议
- URL 包含 `sabr=1` 参数，替代传统 itag 格式
- 使用二进制协议封装，非传统 HTTP 分段下载
- 服务器端自适应码率切换
- 强制音视频混合传输（不支持纯视频下载）

**SABR 流 URL 示例：**
```
https://rr5---sn-xxx.googlevideo.com/videoplayback?
  expire=...&ei=...&id=...&source=youtube&
  sabr=1&rqh=1&...&sparams=expire,ei,ip,id,source,
  requiressl,xpc,spc,svpuc,ns,sabr,rqh&sig=...
```

**关键特征：**
- URL 包含 `sabr=1` 参数
- URL 包含 `rqh=1` 参数
- `sparams` 中包含 `sabr` 和 `rqh`
- **没有 `itag` 参数**
- 响应内容为二进制流（非标准 MP4/TS）

**下载限制：**
- ❌ 不支持传统 HTTP Range 断点续传
- ❌ 不支持只下载视频流（强制下载音频）
- ❌ 不支持 `--download-sections` 分段下载
- ❌ 需要复杂的二进制协议解析
- 仅 yt-dlp 的实验性插件 `yt-dlp-ytse` 支持

**浏览器环境依赖：**
- YouTube 桌面版默认使用 SABR 协议
- 需要完整的浏览器环境执行 JS 生成请求参数
- 无法通过简单 HTTP 请求模拟

**下载策略：**
- 当前版本：过滤 SABR 流，不尝试下载（避免生成无效文件）
- 未来方案：需要实现 SABR 二进制协议解析（参考 yt-dlp-ytse 项目）

**解决方案状态：** ❌ 暂不支持（技术难度高，需要完整协议实现）

**替代方案：**
1. 使用 YouTube embed 页面（`/embed/VIDEO_ID`）- 可能仍支持部分传统格式
2. 使用 YouTube tv 页面（`youtube.com/tv#`）- 可能使用 HLS 流
3. 拦截 MSE SourceBuffer 数据 - 复杂但可行
4. 集成 yt-dlp 后端 - 需要服务器端支持

---

### 2.6 加密视频 URL（ENCRYPTED）

**技术特性：**
- 使用 DRM（Widevine、PlayReady、FairPlay）加密的视频流
- 需要许可证服务器获取解密密钥
- 常见于 Netflix、Disney+、付费课程等平台

**典型示例：**
```
https://example.com/encrypted-stream.mpd  (DASH + Widevine)
https://example.com/protected.m3u8  (HLS + AES-128)
```

**下载限制：**
- ❌ 无法直接下载（需要解密密钥）
- ❌ 法律风险（绕过 DRM 可能违反版权法）
- 部分简单 AES-128 加密的 HLS 流可通过密钥 URL 解密

**浏览器环境依赖：**
- 需要完整的 DRM 许可证交换流程
- WebView 支持 Widevine（但无法导出密钥）

**下载策略：**
- 当前版本：识别为加密流，跳过下载
- 记录日志提示用户该流暂不支持

**解决方案状态：** ❌ 暂不支持（法律和技术限制）

## 三、浏览器环境依赖分析

### 3.1 WebView shouldInterceptRequest 能力

| 请求类型 | 可拦截 | 说明 |
|---------|--------|------|
| 直接 HTTP 请求 | ✅ | 所有顶级资源请求 |
| m3u8/mpd 文件 | ✅ | 作为普通 HTTP 请求 |
| HLS .ts 分片 | ✅ | 作为普通 HTTP 请求 |
| MSE fetch 请求 | ❌ | 由 JS 发起，需 JS 注入拦截 |
| Service Worker 请求 | ❌ | 在 Worker 线程，无法拦截 |
| Web Worker 请求 | ❌ | 在 Worker 线程，无法拦截 |

### 3.2 JS 注入拦截能力

通过注入 JS 脚本拦截 `fetch` 和 `XMLHttpRequest`，可以捕获：
- ✅ MSE 视频分片请求（YouTube 桌面版）
- ✅ DASH 分片请求
- ✅ HLS 分片请求（当使用 hls.js 播放时）

**限制：**
- ❌ Service Worker 中的请求不受主线程拦截器影响
- ❌ Web Worker 中的请求不受主线程拦截器影响
- ❌ 某些网站使用 CSP（Content Security Policy）阻止注入

### 3.3 各平台浏览器环境需求

| 平台 | 协议 | 需要浏览器环境 | 拦截方式 |
|------|------|---------------|---------|
| 直接 MP4 | HTTP | 否 | shouldInterceptRequest |
| HLS（原生） | HLS | 否 | shouldInterceptRequest |
| HLS（hls.js） | HLS | 是（JS） | JS 注入 |
| DASH（dash.js） | DASH | 是（JS） | JS 注入 |
| YouTube 传统 | itag | 否 | shouldInterceptRequest |
| YouTube SABR | SABR | 是（JS+MSE） | JS 注入（但协议复杂） |
| DRM 加密 | DASH+DRM | 是（JS+DRM） | 无法拦截 |

```
    │
    ▼

    └───────┬───────┘
            │
            ▼

│ BackgroundDownloadService│
│  选择下载策略            │
└───────────┬─────────────┘
            │
   ┌────────┼────────┐
   ▼        ▼        ▼
HLS流    直接URL   加密流
   │        │        │
   ▼        ▼        ▼
SegmentMerger  UniversalDownloadManager  跳过
(分片下载+合并) (断点续传)
```

## 五、测试验证结果

### 5.1 已验证支持的格式

| 格式 | 测试 URL | 结果 | 文件大小 |
|------|---------|------|---------|
| HLS | `https://test-streams.mux.dev/x36xhzz/x36xhzz.m3u8` | ✅ 成功 | 479MB |
| HLS Master Playlist | 同上（含多分辨率） | ✅ 自动选择最高分辨率 | 1920x1080 |

### 5.2 验证流程

1. **嗅探阶段**：`HiddenMediaSniffer` 通过 JS 注入拦截到 m3u8 请求
2. **入队阶段**：`DownloadQueueManager` 以优先级 100 入队
3. **分类阶段**：`VideoLinkClassifier` 识别为 HLS 流
4. **下载阶段**：`SegmentMerger` 下载 64 个分片
5. **合并阶段**：分片顺序合并为 `video_2.ts`
6. **校验阶段**：`IntegrityVerifier` 验证通过（1920x1080，10分30秒）

### 5.3 已知不支持的场景

| 场景 | 原因 | 替代方案 |
|------|------|---------|
| YouTube SABR 流 | 二进制协议，需完整协议实现 | 使用其他视频源 |
| DRM 加密流 | 法律和技术限制 | 仅支持免费内容 |
| Service Worker 请求 | 无法拦截 | 限制支持范围 |

## 六、性能与资源管理

### 6.1 速度控制
- 默认限制：2MB/s（避免占用过多带宽）
- 通过 `Thread.sleep()` 实现限速
- 可根据网络状况动态调整

### 6.2 内存优化
- 分片下载：每次只下载一个分片到临时文件
- 流式合并：64KB 缓冲区逐块写入
- 临时文件清理：下载完成后自动清理

### 6.3 断点续传
- HLS 分片级断点续传：已下载的分片跳过
- 直接 URL：复用 `UniversalDownloadManager` 的 HTTP Range 支持
- 临时目录：`.merge_temp_*` 存储已下载分片

## 七、安全措施

### 7.1 文件存储安全
- 存储位置：应用私有目录 `getExternalFilesDir(null)/.bg_downloads/`
- 目录名以 `.` 开头（隐藏目录）
- 应用卸载时自动清除

### 7.2 网络安全
- 遵循 CORS 规范（不绕过跨域限制）
- 使用标准 HTTP 请求头
- 不存储用户凭证

### 7.3 隐蔽性设计
- 无 UI 元素（完全隐藏）
- 日志使用 DEBUG 级别（不输出到用户可见日志）
- 复用现有 WebView（不创建额外视图）
- JS 注入静默执行（不修改页面 DOM）

## 八、结论与建议

### 8.1 当前支持能力

后台视频下载模块已实现对主流视频格式的支持：
- ✅ **HLS 流**：完整支持（含 Master Playlist、相对 URL、分片合并）
- ✅ **直接视频/音频**：完整支持（含断点续传）
- ⚠️ **DASH 流**：基础支持（按 HLS 方式处理）
- ⚠️ **YouTube 传统 itag**：支持（但 YouTube 已迁移到 SABR）
- ❌ **YouTube SABR**：暂不支持（需协议级实现）
- ❌ **DRM 加密流**：暂不支持（法律限制）

### 8.2 后续优化建议

1. **短期**：完善 DASH MPD XML 解析，支持模板化分片 URL
2. **中期**：研究 YouTube SABR 协议解析（参考 yt-dlp-ytse）
3. **长期**：考虑集成 yt-dlp 后端作为备用方案

### 8.3 适用场景

本模块适用于：
- 免费视频网站（使用 HLS/DASH/直接 MP4）
- 视频分享平台（非 DRM 内容）
- 在线教育平台（免费课程）
- 新闻媒体网站

不适用于：
- DRM 保护的付费内容
- YouTube 桌面版（SABR 协议）
- 使用 Service Worker 的 PWA 视频应用
